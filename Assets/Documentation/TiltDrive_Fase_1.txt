TILTDRIVE — DOCUMENTACIÓN TÉCNICA
FASE 1: ARQUITECTURA DE ESTADO, STORE Y CONFIGURACIÓN BASE DEL VEHÍCULO

---

1. OBJETIVO DE LA FASE 1

Construir la primera base de arquitectura de datos del proyecto TiltDrive, separando correctamente:

* la configuración del vehículo
* el estado actual del vehículo
* el sistema que administra ese estado

La intención principal de esta fase es evitar que la lógica del vehículo quede acoplada desde el inicio a scripts únicos o a variables sueltas dentro de componentes aislados.

Esta fase prepara el proyecto para una evolución ordenada hacia:

* control de vehículo
* lectura de inputs
* HUD
* telemetría
* integración con hardware
* eventos de simulación
* futura expansión a cámara, networking o VR

---

2. MOTIVACIÓN DE ESTA FASE

TiltDrive no será un proyecto simple de movimiento directo.
El vehículo va a requerir múltiples sistemas que interactúan al mismo tiempo, por ejemplo:

* dirección
* aceleración
* frenado
* clutch
* cambio de marcha
* freno de mano
* luces
* control de cámara
* RPM
* torque
* par motor
* velocidad
* comportamiento de transmisión
* comportamiento del motor

Si todos estos parámetros se implementan desde el comienzo dentro de un único `VehicleController`, el proyecto se vuelve difícil de mantener, depurar y ampliar.

Por esa razón se decidió establecer una arquitectura basada en tres bloques:

* **Config**
* **State**
* **Store**

---

3. ESTRUCTURA CREADA EN ESTA FASE

Dentro de:

```text
Assets/_TiltDrive/Scripts/Core
```

se crearon las siguientes carpetas:

```text
State
Store
Config
```

Quedando así:

```text
Assets/
└── _TiltDrive/
    └── Scripts/
        └── Core/
            ├── Config/
            ├── State/
            ├── Store/
            └── GameManager.cs
```

INTENCIÓN:
Separar desde la estructura física del proyecto las responsabilidades principales del flujo de datos.

---

4. BLOQUE 1 — CONFIG

4.1 ARCHIVO CREADO

```text
Assets/_TiltDrive/Scripts/Core/Config/VehicleConfig.cs
```

4.2 TIPO DE IMPLEMENTACIÓN

Se implementó como `ScriptableObject`.

4.3 FUNCIÓN PRINCIPAL

`VehicleConfig` será el contenedor de parámetros base del vehículo.
No representa el estado actual del vehículo, sino sus valores configurables de referencia.

4.4 PARÁMETROS INICIALES DEFINIDOS

### Motor

* rpmMinima
* rpmMaxima
* rpmRalenti

### Velocidad

* velocidadMaximaKmh
* aceleracionBase
* frenadoBase

### Dirección

* anguloDireccionMaximo
* sensibilidadDireccion

### Transmisión

* marchaInicial
* marchaMaxima
* tieneReversa

4.5 ASSET CREADO

```text
Assets/_TiltDrive/ScriptableObjects/Config/VehicleConfig_Default.asset
```

4.6 INTENCIÓN DE ESTE BLOQUE

Evitar valores quemados dentro de scripts futuros.

Esto permitirá que más adelante se puedan cambiar parámetros del vehículo sin modificar código, por ejemplo:

* sensibilidad de dirección
* revoluciones máximas
* velocidad máxima
* fuerza de frenado
* cantidad de marchas

También facilitará crear variantes de vehículos a futuro usando distintos assets de configuración.

---

5. BLOQUE 2 — STATE

5.1 ARCHIVO CREADO

```text
Assets/_TiltDrive/Scripts/Core/State/TiltDriveState.cs
```

5.2 FUNCIÓN PRINCIPAL

Representar el estado actual del vehículo de manera centralizada, organizada y modular.

Este archivo no ejecuta lógica.
Su función es contener y estructurar los datos que representan lo que el vehículo está haciendo o siendo en tiempo real.

5.3 DECISIÓN DE DISEÑO IMPORTANTE

Se decidió no usar un estado plano con muchas variables al mismo nivel, porque eso produce desorden cuando el proyecto crece.

En lugar de eso, el estado fue reorganizado por categorías funcionales.

---

6. ESTRUCTURA MODULAR DEL ESTADO

La estructura propuesta para `TiltDriveState` quedó organizada por módulos:

```text
TiltDriveState
├── Control
├── Engine
├── Transmission
├── Movement
└── Other
```

Cada módulo contiene responsabilidades concretas.

---

7. MÓDULO CONTROL

Este módulo agrupa los parámetros relacionados con la intención del usuario o del hardware de control.

### Parámetros considerados

* dirección
* acelerador
* freno
* clutch
* freno de mano
* luces
* cámara X
* cámara Y
* cámara Z

### Intención

Separar los inputs del usuario del resultado físico del vehículo.

Ejemplo:

* el usuario puede presionar acelerador
* pero eso no significa automáticamente que la velocidad ya cambió

La velocidad pertenece al estado físico, no al input.

---

8. MÓDULO ENGINE

Este módulo agrupa el estado relacionado con el motor.

### Parámetros considerados

* motorEncendido
* rpm
* torque
* parMotor

### Intención

Centralizar el comportamiento calculado o actual del motor en una sola categoría.

Esto facilitará después:

* lectura desde HUD
* telemetría
* lógica de simulación
* depuración del comportamiento del motor

---

9. MÓDULO TRANSMISSION

Este módulo agrupa los parámetros relacionados con la transmisión.

### Parámetros considerados

* marchaActual
* relacionActual
* tieneReversa

### Intención

Separar el comportamiento de transmisión del bloque de motor y del bloque de control.

Esto permitirá que a futuro se puedan manejar mejor sistemas como:

* cambio manual
* caja automática
* relaciones por marcha
* reversa
* validaciones de embrague

---

10. MÓDULO MOVEMENT

Este módulo representa el resultado del movimiento del vehículo.

### Parámetros considerados

* velocidadKmh
* direccionActual

### Intención

Separar el estado cinemático o de desplazamiento del resto del sistema.

Esto es útil porque:

* HUD puede leer este bloque directamente
* física puede escribir aquí
* telemetría puede monitorear este bloque
* cámara o efectos pueden reaccionar según la velocidad

---

11. MÓDULO OTHER

Este módulo queda preparado para expansión futura.

### Uso previsto

Servirá para almacenar parámetros no clasificados aún o nuevas categorías que puedan aparecer posteriormente.

Ejemplos futuros posibles:

* estado de ayudas electrónicas
* estado de sensores
* variables temporales de simulación
* banderas especiales del vehículo

### Intención

Evitar romper la estructura principal cuando el proyecto crezca.

---

12. BLOQUE 3 — STORE

12.1 ARCHIVO CREADO

```text
Assets/_TiltDrive/Scripts/Core/Store/TiltDriveStore.cs
```

12.2 FUNCIÓN PRINCIPAL

Administrar una instancia central de `TiltDriveState`.

12.3 COMPORTAMIENTO IMPLEMENTADO EN ESTA FASE

El `Store` actual:

* crea una nueva instancia del estado
* la guarda como estado actual
* expone ese estado mediante una propiedad pública de solo lectura
* inicializa el estado al arrancar

12.4 ESTRUCTURA FUNCIONAL ACTUAL

### Campo interno

* `estadoActual`

### Propiedad pública

* `EstadoActual`

### Método de inicialización

* `InicializarEstado()`

### Evento actual de vida

* `Awake()`

12.5 INTENCIÓN DEL STORE

El Store será el punto central de lectura y escritura del estado del vehículo.

En esta fase todavía es un Store base, pero deja preparado el camino para futuras funciones como:

* actualizar solo una parte del estado
* despachar acciones
* emitir eventos de cambio
* centralizar mutaciones
* desacoplar HUD, Input, Física y Telemetría

---

13. RELACIÓN ENTRE CONFIG, STATE Y STORE

La arquitectura definida en esta fase sigue esta lógica:

### Config

Contiene parámetros base del vehículo.

Pregunta que responde:
**“Cómo está configurado este vehículo?”**

---

### State

Contiene el estado actual del vehículo.

Pregunta que responde:
**“Qué está ocurriendo en este momento?”**

---

### Store

Administra el estado central.

Pregunta que responde:
**“Dónde se guarda y desde dónde se controla el estado?”**

---

14. VENTAJAS DE ESTA DECISIÓN ARQUITECTÓNICA

La separación realizada en esta fase aporta beneficios directos:

### Organización

Cada responsabilidad queda separada.

### Escalabilidad

Se puede crecer sin reescribir toda la base.

### Mantenibilidad

Es más fácil ubicar errores o modificar sistemas.

### Desacoplamiento

Input, HUD, Motor, Transmisión y Telemetría no dependen todos del mismo script monolítico.

### Reutilización

Se podrán crear distintos vehículos con diferentes configs.

### Compatibilidad futura

La arquitectura facilita:

* simulación más compleja
* networking
* VR
* control por hardware
* depuración avanzada

---

16. QUÉ NO SE IMPLEMENTÓ TODAVÍA EN ESTA FASE

Esta fase todavía no incluye:

* acciones del Store
* eventos de actualización
* listeners del estado
* conexión entre Store y GameManager
* lectura de input real
* lógica de motor
* lógica de transmisión
* actualización física del vehículo
* HUD consumiendo el estado
* control real de cámara

Esto fue intencional, porque primero era necesario construir la base estructural.
